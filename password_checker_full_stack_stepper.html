Password input

The raw password enters the pipeline. Everything runs client-side up to the HIBP and LLM calls.

Where it runs
Browser (JS)React / Vue component
Process
User types into a password input field. An onChange handler fires on every keystroke and passes the raw string downstream. The password never leaves the browser until the HIBP prefix is sent and the LLM call is made.
Output passed downstream
{ password: "P@ssword123!" }

Layer 1 — zxcvbn

Rule-based pattern detection. Runs entirely in the browser, zero latency.

Tool
zxcvbn (npm) dropbox/zxcvbn ~400KB bundle
Process
zxcvbn analyses the password against a large dictionary of common words, names, and passwords. It detects keyboard walks (qwerty), l33t substitutions (p@ss → pass), date patterns, repeats, and sequences. It then estimates how many guesses a real attacker would need using a realistic cracking model.
Key outputs
score: 1 // 0–4 guesses: 24000 guesses_log10: 4.38 crack_time: "3 minutes"
feedback: { warning: "This is a top-10 password", suggestions: [ "Add more words" ] }
What it catches that rules miss
"P@ssword123!" passes every checklist rule (uppercase, digit, special char, 12 chars) but zxcvbn scores it 1/4 because it recognises the l33t pattern and common structure.

Layer 2 — pattern extraction

Hand-engineered features built from the raw password + zxcvbn output. These become the XGBoost feature vector.

Tool
Custom JS (browser) or Python (server)
Features computed
From raw password
length: 12 has_uppercase: 1 has_lowercase: 1 has_digits: 1 has_special: 1 unique_char_ratio: 0.75 char_class_count: 4 entropy_bits: 65.4
From zxcvbn
zxcvbn_score: 1 zxcvbn_guesses_log10: 4.38 has_dictionary_match: 1 has_l33t_sub: 1 has_keyboard_pattern: 0 has_date_pattern: 0 sequence_length: 2
Output — feature vector
A flat array of ~15 numeric values passed directly into the XGBoost model. This is the bridge between the rule layer and the ML layer.

Layer 3 — XGBoost model

Trained ML model. Outputs a calibrated strength score 0–100 based on learned cracking behaviour.

Training tools
XGBoost (Python) scikit-learn pandas + numpy Google Colab (GPU) Weights & Biases
Training process
1. Load RockYou + breach corpus (~14M passwords). 2. Run each through zxcvbn + feature extractor to build the training matrix. 3. Generate labels: use Hashcat guess counts (log10-scaled to 0–100) as ground truth. 4. Train XGBoost regressor with 5-fold cross-validation. 5. Evaluate MAE against held-out set. 6. Export trained model to ONNX format.
Deployment tools
ONNX export ONNX Runtime Web (browser) or FastAPI endpoint (server)
Output
{ strengthScore: 23, confidence: 0.91 }
This score is grounded in real cracking data — not rule compliance. "P@ssword123!" scores 23/100 despite passing all policy rules.

Layer 4 — HIBP breach check

Checks whether this exact password has appeared in a known data breach. Uses k-anonymity — only 5 hash chars leave the browser.

Tools
Web Crypto API (SHA-1) HIBP /range endpoint fetch (browser)
Process
SHA-1 hash computed locally → first 5 chars sent to HIBP → ~500 suffix matches returned → local comparison to check if full hash suffix is in the list. The API never sees the full hash or password.
Output
{ isBreached: true, breachCount: 52138 }

Layer 5 — LLM explanation (Claude)

Claude receives all prior outputs as a structured prompt and generates plain-English advice. It does no analysis itself — it only communicates what the other layers found.

Tool
Claude API (claude-sonnet-4-20250514) Streaming response Structured JSON prompt
Prompt payload sent to Claude
{ zxcvbn: { score: 1, warning: "This is a top-10 password", crack_time: "3 minutes", patterns: ["dictionary_match", "l33t_sub"] }, ml_score: 23, policy: { rulesPassed: 11, rulesTotal: 11 }, breach: { isBreached: true, breachCount: 52138 } }
System prompt instruction
You are a password security advisor. Given the analysis below, explain in 2–3 plain sentences why this password is weak and give one specific, actionable suggestion. Do not repeat the raw numbers. Be direct but not alarming.
Output (streamed)
"This password looks complex but follows a pattern attackers know well — replacing letters with symbols like @ and adding numbers at the end. It also appears in over 52,000 known data breaches. Try a random passphrase of 4 unrelated words instead, like 'coral-bench-lamp-seven'."

Final output to user

All four layers converge into a single unified response shown in the UI.

UI components
Strength bar (ML score) Crack time (zxcvbn) Rule badges (policy) Breach warning (HIBP) LLM advice card
Full output object
{ strengthScore: 23, // from XGBoost strengthLevel: "Weak", crackTime: "3 minutes", // from zxcvbn zxcvbnScore: 1, warning: "Top-10 password", // from zxcvbn rulesPassed: 11, // from policy layer rulesTotal: 11, isBreached: true, // from HIBP breachCount: 52138, advice: "This password..." // from Claude (streamed) }
The ML score drives the strength bar. zxcvbn drives the crack time estimate. Policy rules drive the checklist badges. HIBP drives the breach banner. Claude drives the human-readable explanation card. Each piece of UI has exactly one authoritative source.
